> 更新日期：2026-04-08
> 项目：基于 Next.js 的全栈个人知识库问答系统

## 一、项目概述

**一句话描述**：一个基于 RAG 架构的智能知识库问答系统，支持 PDF 上传、语义检索、智能问答和精准引用溯源。

**技术栈**：Next.js 16 + React 19 + TypeScript + PostgreSQL + pgvector + 智谱 AI(GLM-5 + Embedding-3)

## 六、面试话术准备

### Q: 你们的 RAG 是怎么做的？

**参考答案**：
```
整体分为三个阶段：

1. 索引阶段：PDF 上传后按页解析，生成四种 chunk（body/meta/toc/nav），
   智能重叠分块在句子边界切分保留语义完整性，通过 Embedding-3 生成 256 维向量存入 pgvector

2. 检索阶段（核心亮点）：
   - 第一阶段：用户问题先发给 GLM，分析生成 2-5 个搜索关键词
   - 第二阶段：多关键词并发检索，按 chunkType 分组取不同 top-k
   - 最后按 chunkType 权重合并，去重后注入 prompt

3. 生成阶段：带上下文的 prompt 发给 GLM-5，输出答案用 [[REF]] 标记引用来源
```

### Q: 为什么要做两阶段查询分析？

**参考答案**：
```
主要解决用户问题与知识库术语不一致的问题。

比如用户问"钩子函数是什么"，但知识库里写的是"React Hooks"，
直接检索可能匹配不到。但 GLM 知道它们是同一个概念，
会生成 ["React Hooks", "useState", "useEffect"] 等关键词去检索，
召回率大幅提升。

这个方案比纯语义向量检索更精准，比纯关键词检索更灵活。
```

### Q: 256 维是怎么选的？

**参考答案**：
```
主要考虑三点：
1. 成本：智谱 embedding-3 支持维度裁剪，256 维比默认维度减少 75% 存储
2. 速度：向量维度越低，检索计算越快
3. 精度：对于中文文档问答场景，256 维足够保留语义精度

实际测试中 256 维与更高维度的召回率差异在可接受范围内。
```
